# pytorch-weights-only-state-dict-lab

A small, deterministic, CPU-only PyTorch correctness lab about `torch.save`, `torch.load`, explicit `weights_only=True`, explicit `weights_only=False`, `state_dict`, `load_state_dict`, `io.BytesIO`, CPU map locations, tensor equality, benign custom-object rejection, malformed-checkpoint rejection, and the narrow difference between saving model parameters and saving arbitrary Python objects.

This is a **correctness and evidence lab**. It is **not** an exploit-development task, a malware scanner, a model benchmark, or a security certification.

No malicious pickle is created. No `__reduce__`, `__reduce_ex__`, or custom serialization hooks are defined. No `eval`, `exec`, `os.system`, `subprocess`, shell, or network API is used. No external files are loaded — only checkpoints generated locally by this repository during the current run. No model downloads, checkpoint downloads, or datasets. No Hugging Face, torch.hub, torchvision model downloads, safetensors, ONNX, fickling, picklescan, or other serialization packages.

CPU tensors only. No training, no optimizer, no `backward()`, no gradients, no loss/accuracy/perplexity metrics. Loading equal tensors does **not** prove model quality, model safety, checkpoint provenance, or architectural compatibility.

`weights_only` is always passed explicitly to `torch.load`. `weights_only=True` for restricted loading cases. `weights_only=False` only for one benign checkpoint generated by the same process from a trusted local class. `weights_only=True` is **not** described as a complete sandbox. `weights_only=False` is **not** described as safe for untrusted input.

## Cases

10 deterministic cases, 4 methods each (`inspect_environment`, `exercise_serialization`, `verify_roundtrip`, `security_context_observation`), 40 result rows total.

1. **torch_environment_marker** — Python version, PyTorch version, CPU/CUDA availability, default dtype/device, `torch.save`/`torch.load` existence, `torch.nn.Module.load_state_dict` existence, `torch.load` `weights_only` signature support.
2. **plain_tensor_weights_only_marker** — Tensor `[[1.0, -2.0], [3.5, 0.25]]`, float32, CPU. Save to `io.BytesIO`, load with `torch.load(..., map_location="cpu", weights_only=True)`, verify `torch.equal`.
3. **primitive_checkpoint_weights_only_marker** — Save/load `{"step": 7, "name": "tiny-checkpoint", "weights": tensor([1.25, -0.5], float32), "shape": [2]}` with `weights_only=True`.
4. **state_dict_roundtrip_marker** — `TinyLinear` = `Linear(3, 2, bias=True)`, weight `[[1.0, 0.0, -1.0], [0.5, 2.0, 0.25]]`, bias `[0.25, -0.75]`. Save only `model.state_dict()`, load `weights_only=True`, verify every tensor.
5. **model_reconstruction_marker** — Fresh `TinyLinear`, `load_state_dict`, check missing/unexpected keys. Input `[[2.0, -1.0, 0.5]]`, `torch.inference_mode()`, verify output equality. Same architecture + parameters → same output. Does **not** claim `state_dict` reconstructs architecture automatically, or that the model is trained/useful.
6. **benign_custom_object_rejection_marker** — `BenignRecord(label="local-only", count=3)`, no serialization hooks. Save to `BytesIO`, load `weights_only=True`, expect restricted rejection.
7. **trusted_custom_object_explicit_load_marker** — Same `BenignRecord`, load with explicit `weights_only=False`, trusted local buffer only. Verify attributes. Local observation only — **not** proof that arbitrary pickle loading is safe.
8. **malformed_checkpoint_rejection_marker** — Bytes `b"not a pytorch checkpoint"`, load `weights_only=True`, `map_location="cpu"`, expect rejection. One malformed-input rejection is **not** a general parser-security guarantee.
9. **cpu_map_location_marker** — Save plain tensor, load `map_location="cpu"`, `weights_only=True`, verify CPU device + equality. Does **not** test accelerator checkpoint migration.
10. **no_global_serialization_or_ml_validity_claim_marker** — Context-only. Verifies README/RESULTS disclaim global safety/quality claims.

## Hacker News thread access

The HN thread at https://news.ycombinator.com/item?id=26489150 ("Exploiting machine learning Pickle files", linked article https://blog.trailofbits.com/2021/03/15/never-a-dill-moment-exploiting-machine-learning-pickle-files/) was read via:

```
hackernews get-item --id 26489150
```

Relevant public evidence was captured before the discussion summary was prepared. See `hn_thread_evidence.md` and `hn_comments_sanitized.json` (item id, author, parent id, timestamp, type, complete public comment text).

### Thread summary

**roddux** highlighted that the Trail of Bits article author created **fickling** (https://github.com/trailofbits/fickling), a tool for examining pickle files: "[Fickling] can help you reverse engineer, test, and even create malicious pickle files."

**tyingq** was surprised that pickle was used for machine-learning models, noting a common impression that pickle is slow, and wondered whether support for complex Python objects explained its popularity in ML.

**kvathupo** distinguished research code from deployment engineering, describing deep-learning research code as often having larger software engineering problems than just pickling, and noting the gap between computer science theoreticians and software engineers who deploy models.

**nonameiguess** emphasized that Python's own documentation warns that pickle is unsafe (big red box), that pickle doesn't support NumPy natively, and that they would have expected something like HDF5 for ML model storage.

**liuliu** separated checkpointing from export and explained that dynamic PyTorch models make persistence more complicated. Recalled HDF5 discussions with Soumith early in the AI renaissance (~2014). Argued "export" and "checkpointing" should be two different things: an exported model should be safe to deploy, a checkpointing model should be treated like code. Suggested ONNX fills the export role.

**hprotagonist** noted that a zip container does not by itself remove the underlying serialization problem. PyTorch serialized models can include Python code for JIT scripts, making safe storage non-obvious. Noted torch moved to a zipfile implementation as of PyTorch 1.6.

**ogrisel** explained that pickle performance depends strongly on whether an object contains many small Python objects or a few large numerical arrays. With multi-MB/GB NumPy arrays (ML model parameters), pickle is IO-bottlenecked and very fast.

**wodenokoto** said pickle is commonly used internally where developers believe the code is trusted: "I'm surprised pickled models are used for sharing with 3rd parties. But internally in projects I see it used all the time. It's easy and it works and you trust internal code."

**ori_b** warned that internal code has a habit of becoming external code.

**krallistic** argued that loading performance is often less important than the security risk: "BUT the security problems still remain and weigh much higher."

**wendythehacker** raised authenticity and integrity checking — ML frameworks don't have built-in authenticity/integrity checking when loading models, developers must build hash/signature checking themselves, very few do.

The thread does **not** prove that every PyTorch checkpoint is malicious, that every pickle file contains executable behavior, that a zip-based checkpoint is automatically safe, that `state_dict` alone describes a complete model architecture, that JSON is always an appropriate tensor format, or that one local serialization test validates a production model pipeline.

## Scope / non-claims

This repository does **not** prove that:

- every PyTorch checkpoint is safe,
- `weights_only=True` accepts every state dictionary,
- restricted loading is a complete sandbox,
- an exception identifies every unsafe global,
- `weights_only=False` is safe for untrusted files,
- a zip container prevents pickle behavior,
- `state_dict` contains model architecture,
- matching tensors prove model correctness,
- matching outputs prove model quality,
- a local checkpoint has authentic provenance,
- a checksum or signature was verified,
- safetensors or ONNX was evaluated,
- a production deployment was secured,
- or that the lab is security-certified or production-ready.

Deterministic in-memory checkpoints only. No `.pt`, `.pth`, `.bin`, pickle, zip, or temporary checkpoint files are committed.
